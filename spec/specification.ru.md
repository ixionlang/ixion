## 1. Область применения

Настоящий документ определяет:

- лексические правила Ixion;
- синтаксис программы;
- синтаксис объявлений, выражений и типов;
- формальные ограничения, которые проверяются на этапе парсинга;
- грамматику в нотации EBNF.

Документ не определяет:

- внутренние детали кодогенерации;
- ABI JVM;
- оптимизации;
- полную статическую семантику, если она не следует непосредственно из парсера и лексера.

## 2. Нормативные термины

- "ДОЛЖЕН" означает обязательное требование;
- "НЕ ДОЛЖЕН" означает запрещенное поведение;
- "СЛЕДУЕТ" означает рекомендуемое поведение;
- "МОЖЕТ" означает допустимое, но не обязательное поведение.

## 3. Лексическая структура

### 3.1. Пробельные символы

Лексер ДОЛЖЕН игнорировать следующие символы:

- пробел;
- табуляцию;
- возврат каретки;
- перевод строки.

### 3.2. Комментарии

Поддерживаются два вида комментариев:

- однострочные комментарии `// ...`;
- многострочные комментарии `/* ... */`.

Комментарии ДОЛЖНЫ интерпретироваться как пробельные символы.

### 3.3. Идентификаторы

Идентификатор ДОЛЖЕН начинаться с буквы Unicode или символа `_`.

Последующие символы ДОЛЖНЫ быть буквами Unicode, цифрами или `_`.

Идентификатор НЕ ДОЛЖЕН совпадать с зарезервированным ключевым словом языка.

### 3.4. Зарезервированные слова

Следующие лексемы зарезервированы:

- `case`
- `for`
- `while`
- `if`
- `else`
- `return`
- `struct`
- `enum`
- `def`
- `type`
- `const`
- `var`
- `pub`
- `use`
- `new`
- `lambda`
- `true`
- `false`

### 3.5. Литералы

Поддерживаются следующие виды литералов:

- целые числа;
- числа с плавающей точкой;
- числа с суффиксом `f`;
- числа с суффиксом `d`;
- числа с экспонентой;
- символьные литералы;
- строковые литералы;
- булевы литералы.

### 3.6. Строковые литералы

Строковый литерал ДОЛЖЕН заключаться в двойные кавычки.

Поддерживаются стандартные escape-последовательности Java-совместимого вида, включая:

- `\n`
- `\t`
- `\r`
- `\b`
- `\f`
- `\\`
- `\'`
- `\"`
- `\0`

### 3.7. Символьные литералы

Символьный литерал ДОЛЖЕН заключаться в одинарные кавычки.

Символьный литерал МОЖЕТ содержать один символ или escape-последовательность.

### 3.8. Операторы и разделители

Язык использует следующие операторы и разделители:

- `(`
- `)`
- `{`
- `}`
- `[`
- `]`
- `,`
- `...`
- `..`
- `.`
- `::`
- `:`
- `_`
- `=`
- `>=`
- `<=`
- `>`
- `<`
- `==`
- `!=`
- `&&`
- `||`
- `^`
- `+`
- `*`
- `-`
- `/`
- `%`
- `**`
- `|`
- `=>`
- `!`
- `++`
- `--`

## 4. Синтаксические категории

### 4.1. Программа

Программа ДОЛЖНА состоять из нуля или более операторов верхнего уровня.

### 4.2. Блок

Блок представляет собой последовательность операторов в фигурных скобках.

### 4.3. Оператор

Оператором может быть:

- импорт `use`;
- модифицированное объявление `pub`;
- объявление типа `type`;
- объявление функции `def`;
- объявление переменной/константы `var`/`const`;
- условный оператор `if`;
- оператор цикла `while`;
- оператор цикла `for`;
- оператор сопоставления `case`;
- оператор `return`;
- блок;
- выражение.

## 5. Импорт

### 5.1. Синтаксис

Форма импорта:

```ixion
use <path/to/module>
```

### 5.2. Правила

Импорт ДОЛЖЕН начинаться с `<` и заканчиваться `>`.

Путь внутри угловых скобок ДОЛЖЕН состоять из идентификаторов, разделенных `/`.

Алиасы импорта в текущей реализации НЕ ПОДДЕРЖИВАЮТСЯ.

## 6. Объявления верхнего уровня

### 6.1. `pub`

Ключевое слово `pub` ДОЛЖНО предшествовать только экспортируемому верхнеуровневому объявлению.

Экспортируемыми считаются:

- функция;
- переменная;
- константа;
- структура;
- перечисление;
- type alias.

### 6.2. `type`

Форма объявления типа:

```ixion
type Name = TypeExpression
```

После `type` и имени ДОЛЖЕН следовать `=`.

Правая часть ДОЛЖНА быть одной из следующих форм:

- alias типа;
- `struct`;
- `enum`.

### 7.3. `struct`

Структура объявляется в форме:

```ixion
type Name = struct [T, U, ...] {
    field : Type
}
```

Список generic-параметров является необязательным.

Поля структуры ДОЛЖНЫ задаваться как `name : Type`.

### 6.4. `enum`

Перечисление объявляется в форме:

```ixion
type Name = enum {
    value1,
    value2
}
```

Каждое значение enum ДОЛЖНО быть идентификатором.

### 6.5. `def`

Функция объявляется в форме:

```ixion
def name(a : A, b : B) : R {
    ...
}
```

Список параметров ДОЛЖЕН использовать форму `name : Type`.

Возвращаемый тип является необязательным, но если он указан, тело функции ДОЛЖНО содержать оператор `return` в соответствии с проверкой достижимости, реализованной парсером.

## 7. Переменные

### 7.1. Синтаксис

```ixion
var name = expression
const name = expression
```

### 7.2. Правила

Объявление переменной ДОЛЖНО содержать:

- ключевое слово `var` или `const`;
- имя;
- оператор `=`;
- инициализирующее выражение.

В текущей реализации аннотация типа в объявлении переменной отсутствует на синтаксическом уровне, в дальнейшем планируется реализация и обновление спецификации.

## 8. Управляющие конструкции

### 8.1. `if`

Форма:

```ixion
if condition {
    ...
} else if otherCondition {
    ...
} else {
    ...
}
```

Условие ДОЛЖНО быть выражением.

Скобки вокруг условия МОГУТ присутствовать, если это допускается как выражение.

### 8.2. `while`

Форма:

```ixion
while condition {
    ...
}
```

### 8.3. `for`

Форма:

```ixion
for item : iterable {
    ...
}
```

Цикл `for` представляет собой форму обхода по коллекции или итерируемому выражению.

### 8.4. `case`

Форма:

```ixion
case expression {
    Type name => statement-or-block
}
```

Каждая ветвь `case` ДОЛЖНА содержать:

- типовую альтернативу;
- имя связываемой переменной;
- оператор `=>`;
- тело ветви.

Тело ветви МОЖЕТ быть либо блоком, либо одиночным оператором.

## 9. Возврат

### 9.1. Синтаксис

```ixion
return
return expression
```

### 9.2. Правила

Если выражение отсутствует, `return` трактуется как пустой возврат.

## 10. Выражения

### 10.1. Общие правила

Выражения строятся с фиксированными уровнями приоритета (парсер Pratt'a).

### 10.2. Приоритеты

От меньшего к большему:

1. присваивание;
2. XOR `^`;
3. OR `||`;
4. AND `&&`;
5. сравнение;
6. сложение и вычитание;
7. умножение, деление и остаток;
8. возведение в степень;
9. префиксные операторы;
10. постфиксные операторы;
11. первичные выражения.

### 10.3. Присваивание

Оператор `=` ДОЛЖЕН интерпретироваться как инфиксный оператор присваивания.

### 10.4. Префиксные операторы

Поддерживаются:

- `+`
- `-`
- `%`
- `!`

### 10.5. Постфиксные операторы

Поддерживаются:

- `++`
- `--`

### 10.6. Вызов

Вызов функции ДОЛЖЕН иметь форму:

```ixion
expr(arg1, arg2)
```

### 10.7. Доступ к свойству

Доступ к свойству ДОЛЖЕН иметь форму:

```ixion
expr.name
```

### 10.8. Индексация

Индексация ДОЛЖНА иметь форму:

```ixion
expr[index]
```

### 10.9. Литерал списка

Литерал списка ДОЛЖЕН иметь форму:

```ixion
[expr1, expr2, expr3]
```

Пустой литерал списка допускается.

### 10.10. Группировка

Группировка ДОЛЖНА использовать круглые скобки:

```ixion
(expression)
```

### 10.11. Лямбда

Лямбда ДОЛЖНА иметь форму:

```ixion
lambda (a : A, b : B) : R {
    ...
}
```

Допускается сокращенная форма с одним параметром без круглых скобок:

```ixion
lambda x : R {
    ...
}
```

В текущей реализации тело лямбды ДОЛЖНО быть блоком.

## 11. Типы

### 11.1. Типовая ссылка

Типовая ссылка ДОЛЖНА начинаться с идентификатора.

Допускаются:

- составные имена через `.`;
- списочные типы через `[]`;
- объединения через `|`.

### 11.2. Списочные типы

Тип `T[]` ДОЛЖЕН обозначать список элементов типа `T`.

### 11.3. Объединения

Тип `A | B | C` ДОЛЖЕН обозначать объединение альтернатив.

### 11.4. Квалифицированные имена

Тип `a.b.c` ДОЛЖЕН трактоваться как квалифицированное имя типа.

## 12. Семантические ограничения, проверяемые парсером

### 12.1. Функции с возвращаемым типом

Если функция объявлена с возвращаемым типом, парсер ДОЛЖЕН проверить наличие `return` в теле функции по доступному синтаксическому анализу.

### 12.2. Экспорт

После `pub` ДОЛЖНО следовать только экспортируемое объявление.

### 12.3. `use`

Импорт ДОЛЖЕН использовать угловые скобки и путь из идентификаторов.

### 12.4. `case`

В каждой ветви `case` ДОЛЖНО быть указано имя связываемой переменной перед `=>`.

## 13. Грамматика

Ниже приведена формальная грамматика в EBNF.

```ebnf
program         = { statement } EOF ;

statement       = use_stmt
                | pub_stmt
                | type_stmt
                | function_def
                | variable_decl
                | if_stmt
                | while_stmt
                | for_stmt
                | case_stmt
                | return_stmt
                | block
                | expr_stmt ;

pub_stmt        = "pub" statement ;

use_stmt        = "use" "<" use_path ">" ;
use_path        = ident { "/" ident } ;

type_stmt       = "type" ident "=" type_rhs ;
type_rhs        = struct_def | enum_def | type_expr ;

struct_def      = "struct" [ "[" ident { "," ident } "]" ] "{" { field_decl } "}" ;
field_decl      = ident ":" type_expr [ "," ] ;

enum_def        = "enum" "{" { enum_value [ "," ] } "}" ;
enum_value      = ident ;

function_def    = "def" ident
                  "(" [ param { "," param } ] ")"
                  [ ":" type_expr ]
                  block ;

param           = ident ":" type_expr ;

variable_decl   = ( "var" | "const" ) ident "=" expression ;

if_stmt         = "if" expression block [ "else" ( if_stmt | block ) ] ;
while_stmt      = "while" expression block ;
for_stmt        = "for" ident ":" expression block ;

case_stmt       = "case" expression "{"
                  { case_branch }
                  "}" ;
case_branch     = type_expr ident "=>" ( block | statement ) ;

return_stmt     = "return" [ expression ] ;

block           = "{" { statement } "}" ;
expr_stmt       = expression ;

expression      = assignment ;
assignment      = or_expr [ "=" assignment ] ;

or_expr         = xor_expr { "||" xor_expr } ;
xor_expr        = and_expr { "^" and_expr } ;
and_expr        = comparison { "&&" comparison } ;
comparison      = sum_expr { ( "==" | "!=" | "<" | ">" | "<=" | ">=" | ".." ) sum_expr } ;
sum_expr        = product_expr { ( "+" | "-" | "%" ) product_expr } ;
product_expr    = exponent_expr { ( "*" | "/" ) exponent_expr } ;
exponent_expr   = prefix_expr [ "**" exponent_expr ] ;
prefix_expr     = { ( "+" | "-" | "%" | "!" ) } postfix_expr ;
postfix_expr    = primary_expr { postfix_op } ;
postfix_op      = "++" | "--" | call_suffix | index_suffix | property_suffix ;
call_suffix     = "(" [ expression { "," expression } ] ")" ;
index_suffix    = "[" expression "]" ;
property_suffix = "." ident ;

primary_expr    = literal
                | ident
                | "(" expression ")"
                | list_literal
                | lambda_expr ;

list_literal    = "[" [ expression { "," expression } ] "]" ;

lambda_expr     = "lambda" ( "(" [ param { "," param } ] ")" | ident )
                  ":" type_expr
                  block ;

type_expr       = type_union ;
type_union      = type_ref { "|" type_ref } ;
type_ref        = ident [ "[]" ] [ "." type_ref ] ;

literal         = int_lit
                | float_lit
                | double_lit
                | char_lit
                | string_lit
                | "true"
                | "false" ;

ident           = IDENTIFIER ;
EOF             = end_of_file ;
```

## 14. Соответствие текущей реализации

Документ отражает следующие факты текущей реализации:

- лексер поддерживает комментарии, строки, символы и числа с суффиксами;
- объявление функций, структур, enum, алиасов и импортов поддерживается;
- `case` реализован как pattern matching по типу;
- `lambda` реализована как блоковая конструкция;
- списковые типы и объединения типов поддерживаются на уровне синтаксиса.
